## 好的部分

### 1. 架构设计清晰
- 三类节点（Processor/Trigger/Listener）的抽象合理，覆盖了视频工厂未来的扩展方向（定时采集、IM 通道触发）
- 配置与连线分离（`config_schema` vs `inputs/outputs`）解决了旧体系里前端配置项和连线桩混淆的问题
- 生命周期钩子 `prepare -> validate -> check_cache -> execute -> on_error -> finalize` 完整且实用

### 2. 向后兼容做得好
- 保留了旧 `reads/writes`、`@register` 装饰器、`output_dirs` 缓存检查
- `get_definition()` 自动从旧 `reads/writes` 生成 `inputs/outputs` 连线桩
- 不强制迁移，新旧节点可以共存

### 3. Executor 实现扎实
- 基于 edges 的拓扑排序，连线决定执行顺序
- 同层 `asyncio.gather` 并发
- `resume_from_node`、`dry_run`、`force_no_cache` 都是实际有用的功能
- retry 逻辑、`on_failure=skip/abort` 策略合理
- 子进程终止（psutil kill children）处理了停止执行时下载卡住的问题

### 4. 节点包体系前瞻性好
- `vf-node.yaml` 清单 + `discover_node_packs` 自动加载，后续社区扩展有基础
- 版本管理 + `migrate_config` 配置迁移

### 5. 文档质量高
- `CUSTOM_NODES.md` 开发者友好，有完整示例代码
- `NODE_SYSTEM_RUNTIME.md` 实事求是记录了已实现 / 未验证 / 已知限制

## 问题和风险

### ~~已排除：_terminate_subprocesses 杀全部子进程太粗暴~~
- **原始评价**: `psutil.Process.children(recursive=True)` 会杀掉当前进程的所有子进程，包括 uvicorn 自身的 worker
- **用户确认**: 这是 Ctrl+C 时才触发的，杀全部子进程是合理的
- **结论**: 不是问题，已排除

### 高优先级

#### 1. on_progress 只写内存不推 WebSocket
- **位置**: `server/engine/executor.py` `_run_node()` 中的 `on_progress` 回调
- **问题**: `on_progress` 只更新了 `NodeState` 的内存状态，没有推送到前端 WebSocket
- **影响**: 前端看不到实时进度条变化，除非有日志触发 drain
- **修复方向**: 在 `on_progress` 回调里加 `await self._emit("node_progress", {...})`，或把进度信息塞进 `log_queue`

### 中优先级

#### 2. WebSocketLogHandler 并发时日志重复
- **位置**: `server/engine/executor.py` `WebSocketLogHandler`
- **问题**: 同层 3 个节点并发时，`self._current_nodes` 列表有 3 个 node_id，每条日志会被标记为 3 个 node_id 发送 3 次
- **影响**: 前端日志重复
- **修复方向**: 给每个 logger 绑定单一 node_id，或者用 `extra` 字段传递 node_id 而不是用全局列表

#### 3. 内置节点还在用旧 reads/writes，edge binding 实际不生效
- **位置**: `server/engine/executor.py` `_inject_edge_bindings()`
- **问题**: 旧节点没有 `inputs` 声明（用 `reads`），edge binding 注入会静默跳过
- **影响**: 当前内置节点大部分还是旧方式，连线的 `source_handle`/`target_handle` 实际不起作用
- **修复方向**: 要么把内置节点迁移到新 `inputs/outputs` 声明，要么在 `_inject_edge_bindings` 里对旧节点做兼容处理

### 低优先级

#### 4. pyyaml 未加 requirements.txt
- **问题**: `discover_node_packs` 依赖 `yaml` 但 `requirements.txt` 只有 28 bytes
- **影响**: `import yaml` 如果没装 pyyaml 会直接崩；这个模块是按需 import 的，初次启动不一定触发，容易被遗漏
- **修复方向**: 把 `pyyaml` 加到 `requirements.txt`

#### 5. send_reply 读 reply 的语义有问题
- **位置**: `server/engine/executor.py` `_run_listener()`
- **问题**: `ctx.read(trigger.id, "reply")` 语义是读 trigger 自己的 output，但 reply 应该来自下游节点
- **影响**: Listener 模式下拿不到正确的回复内容
- **修复方向**: 应该扫描下游节点的 outputs，而不是读 trigger 自己的 output

#### 6. 没有处理 sys.path 问题
- **问题**: 和 main 分支一样的问题——节点 `execute()` 里 `import agents.*` 时在 uvicorn reload 子进程中会找不到模块
- **修复方向**: 在 `run_server.py` 或 `executor.py` 开头统一处理 `sys.path`

#### 7. 缓存 fingerprint 文档不完整
- **问题**: `cacheable` 默认 `False`（好的设计决定，LLM 不幂等），但文档里没有明确说明哪些内置节点开了缓存
- **修复方向**: 在 `NODE_SYSTEM_RUNTIME.md` 里补充缓存节点清单

## 总结

整体来说架构方向对、文档好、兼容性考虑周全，但有几个运行时细节会在实际跑流水线时出问题。建议先修高优的 `on_progress` 不推 WebSocket 再合并。
